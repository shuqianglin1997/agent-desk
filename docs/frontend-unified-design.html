Frontend Unification Plan

箱猫猫庭院 × 经典视图 · 前端统一设计 一套骨架，两种呈现 —— 把庭院做成一格定比的箱庭

现在两个模式是两套互不相干的布局，靠改 CSS 各自硬撑；账号被画了三遍（侧栏列表 / 庭院 chips / 庭院猫），场景没有确定比例，一拉宽就空。本规划把它收敛成一套区域骨架 + 一个可替换的「账号呈现层」：猫猫庭院按电影屏幕比例做成定格居中的箱庭，自适应按业内标准（宽高比锁定 + 断点 + 流式缩放）设计。

1·3 共享的其实是数据，不是布局已经统一的部分

两个模式底下是同一个 store。账号、会话、活跃状态、额度都只有一份；上层把「账号名册」画了三遍——侧栏列表、庭院 chips、庭院猫，全从同一份 state.profiles 派生。所以真正分叉的是布局骨架和呈现，数据层早就是一源。

为什么「感觉不行」——四个病灶

① 两套骨架。经典=侧栏+竖堆，庭院=横带+分屏。改一处要跟另一套 CSS 引擎打架（本轮就是不断在 body[data-view=yard] 上叠覆盖）。
② 账号画三遍。列表/chips/猫 三套呈现各写各的，样式与交互难一致。
③ 场景没有比例。要么定死一个窄块，要么拉成全宽——全宽后道具聚左、右半空旷。缺一个「确定的画框」。
④ 没有统一自适应。两视图各写一堆 @media，断点口径不一。
合起来：像两个共享数据的 App，不像一个产品的两种皮肤。

弐

设计原则：最高标准 · 日本产品

Principles

「统一 + 最高标准 + 参考日本产品」不是贴风格，是一套取舍纪律。下面每条都对应一个具体决定。

箱庭 · Hakoniwa

日本的箱庭是一个定尺的盒子里的微缩庭院。把猫猫庭院当箱庭：一个固定比例的画框，像额縁（画框）一样把景致框住、居中呈现——而不是把景致抻满整条屏。比例是它的第一属性。

間 · Ma（留白）

「引き算の美学」——减法。空不是没填满，是刻意的呼吸。场景定比后两侧的余量不是 bug，是画框的卡纸（passe-partout）。删掉账号的三遍呈现，只留最合适的一种。

具体参照

無印良品减法、无品牌的平静。「これでいい」——够用即最好 → 砍掉冗余呈现，一套克制骨架。
任天堂约束内的趣味与精度 → 猫的憨与生动保留，但装进精确的画框里。
深澤直人without thought——顺手即对 → 布局层级要「不用想就懂」：选账号→看会话→取交接。
額縁 / 盆栽框与比例的美学 → 场景=定比画框；克制、定尺、居中。
一句话纪律

凡是要「各写一套」的冲动，先问：能不能是同一套骨架的一个参数？凡是要「填满」的冲动，先问：比例对不对？

参

统一架构：一套骨架，账号呈现层可换

One Skeleton

核心动作只有一个：定义一套区域骨架，把两个模式唯一真正不同的东西——「你怎么看到并挑选账号」——抽成一个可替换的呈现槽。其余全共享。

3·1 区域骨架（两模式共用）

自上而下一条清晰的动线：挑账号 → 该账号的操作 → 它的会话 → 某条会话的交接。这是标准的主从（master–detail）结构，Mail.app 之类的高标准应用都用它。

3·3 账号：从「画三遍」收敛成「一个呈现层，两种实现」

把侧栏列表 / chips / 猫 收成 AccountPresenter 接口的两个实现：ScenerPresenter（庭院·猫）与 RosterPresenter（经典·名册）。它们吃同一份账号投影、发同一个「选中账号」事件。chips 快切条并入场景本身（场景已经能点选，不再另立一条）。经典不再需要独立侧栏——名册即呈现层。

对应到架构纪律

这正是 editor-architecture 的「Layout Is Data / Registry-Driven / Selection Drives Content」：呈现层按注册表选择，选中态驱动下游，面板之间只经 store 与事件通信。加新呈现方式（比如「紧凑列表」）= 加一个 Presenter 实现，不动骨架。

肆

猫猫庭院比例：电影屏幕

Cinematic Frame

场景「感觉不行」的根子是没有确定比例。修法：给它一个固定的电影宽高比，像放电影一样等比适配、居中、两侧留边（letterbox / pillarbox）——永远不拉伸、永远不空旷。

4·1 为什么本轮「全宽」是错的

本轮改动①把逻辑宽 W 拉成跟容器走。结果：地形铺满了，但工作亭/池塘/树这些道具是定点构图，一宽就聚在左边、右半只剩草皮；越宽越空。这是"抻画"而不是"取景"。正确的类比不是「把画拉满墙」，是「把画装进合适的画框，挂正中」。

诚实交代

本规划要回退改动①的响应式 W（6e3c574），换成「定比画布 + CSS 等比适配」。改动②的分屏工作区可保留复用（见 §6）。

4·2 选哪个比例

顶部陪伴带要宽而不太高。越宽的比例越矮。候选（越往下越宽越矮）：

推荐 2.39∶1（CinemaScope）：这是「电影宽银幕」最标志性的比例，做顶部横带宽扁得当。逻辑画布定 574 × 240（=2.392）或 480 × 201；场景按这个画框重新构图一次（工作亭偏左、池塘居中、大树偏右、地平线贯穿），构图从此固定、可精修到像素。若实测觉得太矮压不下大树/工作亭，退一档 2∶1（Univisium）——更稳、仍读作电影感。

4·3 像放电影：等比 + 居中 + 卡纸

画框固定比例，在横带里等比缩放到「能放下的最大」，水平居中，两侧用木质卡纸补边（额縁效果）。宽窗口不会把场景抻稀，只是卡纸变宽；窄窗口画框缩小、卡纸消失。这就是视频播放器 / hero 图的 aspect-ratio + object-fit: contain 标准做法。

连带简化

画框定比后，scene.js 回到固定逻辑尺寸——不再需要响应式 W、道具偏移、散点重算、ResizeObserver。构图一次画对、永远对。代码更少、更稳、画风零风险。

伍

自适应规则：业内标准

Responsive

两条腿：场景用宽高比锁定（上面 §4）；工作区用标准断点 + 流式缩放。桌面应用窗口可任意拉伸，规则要覆盖从超宽到窄。

5·1 断点（内容驱动，非设备驱动）

断点按会话表这块主工作面还能不能舒服并排详情来切，而不是猜设备。会话表永远是主角，优先保它的宽度与行数。

5·2 流式缩放（断点之间不跳变）

横带高clamp(140px, 20vh, 224px) —— 跟视口连续变，画框按 2.39 联动算宽
字号 / 间距关键层级用 clamp() 建流式比例尺，不在每个断点手调
会话表min-width: 0 防溢出；列用 minmax()；表体是唯一滚动区
详情 / 控制台定宽夹 clamp(280px, 24vw, 340px)；窄档转下方/抽屉
与已落地的改动②对齐

本轮已提交的分屏（f58ac9a）里「会话表吃满整高、表体唯一滚动区、<900 折回单列」正是这套规则的一部分，可直接并入——断点口径统一成上表即可（900 → 960/720 两档）。
