# 猫猫庭院美术资产交接规范

本文不是让画师复刻当前程序化像素图，而是定义美术资产与产品逻辑之间的合同。当前猫与庭院只承担功能占位和交互验证；正式资产可以重做构图、比例、建筑和角色细节，只要下述身份、状态、锚点与交互语义仍然成立。

如果画师发现区域关系、角色动作或视觉层级不合理，应先提出修改逻辑，不要为了迁就本文而画出违背直觉的画面。修改区域几何时同步更新 `src/yard/interactions.js` 的 `ZONES`；修改角色状态时同步更新 `src/yard/cats.js` 和本文的状态表。美术和逻辑必须一起验收。

## 一、画面坐标与交付原则

- 庭院逻辑画布固定为 `480 × 132`。应用按窗口宽度整数或非整数放大，但渲染必须保持 `image-rendering: pixelated`，不做平滑插值。
- 所有角色位置使用“脚底中心”锚点。趴卧、睡觉、纸箱等宽姿势也要提供同一含义的锚点，切换动作时猫不能瞬移。
- 场景图不写账号名、分组名、天气名等业务文字。文字由 DOM 覆盖层绘制，以便本地化、无障碍和动态更新。
- 场景中可以画木牌、屏幕、信封等空白载体，但不要把“工作亭”“任务道”之类文字烘焙进 PNG。
- 所有源文件同时交付可编辑版本（Aseprite / PSD / Krita 任一）和导出 PNG。PNG 使用 sRGB、透明背景、无边缘半透明脏像素。
- 像素资产不使用抗锯齿、模糊、半像素描边。阴影可以半透明，但轮廓和可点击边缘必须落在整数像素。
- 同一对象的昼夜/天气版本必须保持外轮廓、锚点和占地不变，避免切换环境时碰撞区漂移。

推荐目录：

```text
assets/yard/
  source/                 可编辑源文件，不进入发布包也可以
  scene/                  场景分层 PNG
  facilities/             可交互设施
  cats/                   猫基础动作与覆盖层 atlas
  effects/                天气、提示和小特效
  ui/                     名牌、气泡、九宫格边框
  manifest.json           文件、帧、锚点和层级声明
```

命名统一使用小写英文和短横线，例如 `cat-sit-base.png`、`facility-mailbox-open.png`。不要在文件名中放账号名或版本号。

## 二、场景分层清单

正式场景建议拆成以下层。这样昼夜、天气与设施状态可以独立变化，不需要为每种组合导出一张完整大图。

| 资产 ID | 建议画布 | 层级 | 状态/变体 | 业务用途 |
|---|---:|---:|---|---|
| `scene.sky` | 480×132 | 0 | day / dusk / night | 天空底色、日月；不能包含会挡住猫的前景 |
| `scene.horizon` | 480×132 | 10 | day / dusk / night | 远山、云层基底、远树，允许水平无缝延展 |
| `scene.ground` | 480×132 | 20 | season 可选 | 草地、道路、基础地面纹理 |
| `scene.path` | 480×132 | 25 | normal | 连接工作亭、任务道、池塘、瞭望点；视觉上必须真能走通 |
| `scene.fence` | 480×132 | 30 | normal | 中景栅栏，不得遮挡角色主要活动带 |
| `scene.foreground` | 480×132 | 90 | normal | 仅放可压在猫前面的少量草叶/栏杆，透明背景 |
| `scene.light-mask` | 480×132 | 95 | dusk / night | 窗灯、台灯、池面反光等加色层，不改变设施轮廓 |

逻辑活动带大致为 `y=68…126`。重要设施不能把可走区域切断；装饰物应让出至少 20×20 的猫落点。画师若需要更大的建筑，可调整构图，但必须保留一条从左侧工作区通往右侧瞭望点的清晰路径。

## 三、可交互设施与真实行为

设施不是背景装饰。用户拖猫时才显示命中提示，放下后产生明确业务意图。

| 区域 | 当前逻辑矩形（480×132） | 需要的资产 | 放下后的真实行为 | 设计要求 |
|---|---|---|---|---|
| 工作亭 `workshop` | x 10–108, y 16–78 | 建筑底、桌椅、屏幕、台灯、工作中覆盖层 | 未运行：确认后打开官方 App；已运行：聚焦账号信息 | 至少容纳 3 只工作猫，桌位不能暗示固定账号归属 |
| 邮筒 `mailbox` | x 104–132, y 40–76 | closed / flag-up / letter-fx | 确认后复制当前会话交接信息 | 投递动画 6–10 帧；成功与失败视觉要可区分 |
| 任务道 `queue` | x 50–194, y 78–111 | 路面标记、排队点、等待小牌 | 把当前会话加入内嵌 Agent 待办；运行中会在上一项完成后发送 | 视觉上是通路或候车点，不画成封闭面板；要能看出先后顺序 |
| 池塘 `attention` | x 190–290, y 87–130 | 水面、岸边、涟漪、可选锦鲤 | 打开当前会话详情 | 池塘代表“停下来查看”，不能让人误以为会删除/归档会话 |
| 瞭望点 `remote` | x 354–470, y 34–91 | 高台/终端屏/信号灯 | 打开当前账号的内嵌终端工作台 | 需要一个清晰的“远程/终端”视觉符号，但不使用品牌 Logo |
| 树下草坪 `meadow` | x 288–470, y 88–130 | 树、阴影、草垫、可选长凳 | 保存猫的自定义位置 | 必须保留多个空闲落点，不能用树冠遮住名牌和气泡 |

矩形只是当前命中层，不是要求画师按框画虚线区域。最终画面中不常驻这些框；拖拽时由程序叠加半透明提示。若设施视觉中心与矩形不一致，优先修正 `ZONES`，不要扩大透明点击区去“骗”命中。

## 四、猫角色资产

### 4.1 基础规格

- 建议统一 atlas 单元为 `24 × 24` 逻辑像素，脚底中心锚点建议 `(12, 21)`。宽姿势可使用 32×24 单元，但锚点语义不变。
- 当前占位底图是 `sit 16×16`、`loaf 20×10`、`sleep 16×8`；正式稿不必沿用尺寸，只需在 manifest 声明每帧矩形和锚点。
- 身体、花纹/毛色、项圈、应用吊牌、默认槽位铃铛、配饰分层。不能为每个组合导出一整套动画，否则组合数量会失控。
- 8 种毛色继续支持：橘、三花、奶牛、黑、白、狸花、蓝灰、暹罗。可用索引色 palette swap，也可提供遮罩层。
- 账号应用颜色只出现在小吊牌：Claude 橙、Codex 青、未来 Agent 从 manifest 读取颜色。默认槽位用铃铛，不把品牌标志画在猫身上。

### 4.2 动作清单

| 动作 ID | 建议帧数 / fps | 触发状态 | 必须传达的信息 |
|---|---:|---|---|
| `idle-sit` | 4 帧 / 2fps | 在岗、普通停留 | 呼吸、眨眼，安静但不是睡着 |
| `walk` | 4–6 帧 / 6fps | 去设施、拖放后移动 | 方向可水平翻转；脚底锚点稳定 |
| `work-type` | 4–6 帧 / 6fps | `working` | 正在敲键盘，屏幕/爪子节奏不要过快 |
| `arrive-sign` | 4 帧 / 3fps | `arriving` | 举开工牌向工作亭走，不等于已经运行 |
| `play` | 6–8 帧 / 5fps | `play` | 探索、看鱼或追光点，不能遮住业务提示 |
| `loaf` | 2–4 帧 / 1fps | `rest` | 醒着休息，和 nap 明显不同 |
| `nap` | 4 帧 / 2fps | `nap` | 蜷睡、轻呼吸；Zzz 为独立效果层 |
| `hibernate` | 4 帧 / 1fps | `hibernate` | 纸箱/毯子，仍能认出毛色和身份吊牌 |
| `confused` | 6 帧 / 4fps | 路径错误 | 犹豫/转圈；不能做成搞笑成功态 |
| `lifted` | 2 帧 / 2fps | 鼠标拖起 | 四肢自然下垂或抱起姿势，影子与身体分离 |
| `pet` | 4 帧 / 4fps | 长按/抚摸 | 眯眼、呼噜；不改变账号业务状态 |
| `tired-overlay` | 2–4 帧 | 额度 10–30% | 小汗滴/慢动作，作为覆盖层，不替代基础动作 |
| `exhausted-overlay` | 2–4 帧 | 额度 <10% | 低电量提示，仍允许继续工作 |

每个动作至少交一张关键帧先做逻辑验收，再补全动画。不要在未确认锚点和轮廓时一次画完全部毛色。

### 4.3 配饰覆盖层

首批保留围巾、眼镜、蝴蝶结、草帽和无配饰。覆盖层必须随 `idle-sit`、`walk`、`work-type`、`lifted` 至少四组动作；睡眠姿势若会穿模，可在 manifest 标注 `hiddenOn`。项圈颜色由程序着色，不要只交固定六色位图。

## 五、环境与反馈资产

| 类型 | 资产 | 要求 |
|---|---|---|
| 云 | 3–5 个透明小云 | 可水平循环；昼夜只调色，不改尺寸 |
| 雨 | 2–3 个雨滴/溅水帧 | 程序可继续生成粒子，素材只定义造型 |
| 雪 | 3 个雪花点 | 低对比，不能盖住账号名牌 |
| 萤火虫 | 3 帧 | 仅晴朗夜晚；数量克制 |
| 水面 | 4–6 帧涟漪 | 池塘循环，不能让整片水面高频闪烁 |
| 信封 | 飞行 6–10 帧 | 邮筒交接成功；失败不播放成功动画 |
| 提示符 | `?`、`!`、低电量、音符、Zzz | 独立覆盖层，颜色满足明暗主题对比 |
| 影子 | 椭圆/姿势遮罩 | 拖起时留在地面，放下时回到脚底 |

天气是自动的本地环境节奏，不代表真实气象。不要画温度、城市名、雷达图等会让用户误以为接入了天气服务的元素。

## 六、UI 资产

- 左上角 Logo：保留当前“小橙方块 + 两个白色窗口/眼睛”的方向，交付 SVG 主文件、16/32/64/256/512 PNG、Windows ICO 和 macOS ICNS。小尺寸必须人工像素对齐，不能只缩放大图。
- 猫名牌：推荐九宫格边框，支持 60–150px 动态宽度；选中、hover、错误、低额度只换边框/底色，不换形状。
- 对话气泡：普通、环境、警告、错误四种语义；箭头可左右移动以避开画布边缘。
- 拖放区域提示：程序生成即可；若交美术边框，必须是九宫格，不能固定写区域名称。
- 右侧信息栏、内嵌终端、按钮和表格属于产品 UI，不纳入场景插画 atlas。不要用大面积木纹图片替代可伸缩 CSS 组件。

## 七、Manifest 合同

正式资产接入时新增 `assets/yard/manifest.json`。建议结构：

```json
{
  "version": 1,
  "logicalSize": [480, 132],
  "assets": {
    "scene.sky.day": { "file": "scene/sky-day.png", "layer": 0 },
    "facility.mailbox.closed": {
      "file": "facilities/mailbox.png",
      "frame": [0, 0, 28, 36],
      "anchor": [14, 34],
      "layer": 42
    },
    "cat.idle-sit": {
      "file": "cats/cat-base.png",
      "frameSize": [24, 24],
      "frames": 4,
      "fps": 2,
      "anchor": [12, 21]
    }
  }
}
```

加载失败、manifest 版本不支持或单个帧缺失时，程序必须回退到当前程序化资产，不能让整个庭院空白。manifest 不保存账号数据和业务状态，只描述文件、帧、锚点、层级和可着色区域。

## 八、首批交付顺序

1. 一张 480×132 的构图草图：只验收区域关系、路径、遮挡和空白，不验收细节。
2. 一只猫的灰模 atlas：只验收脚底锚点、动作轮廓、拖起和工作动作。
3. 白天场景完整分层 + 一种猫毛色 + 基础 UI 九宫格。
4. 昼夜和天气变体、其余毛色、配饰、设施状态动画。
5. 全尺寸 Logo 与打包图标。

## 九、验收清单

- 在 1080×660 最小窗口和 1440×900 默认窗口中，猫、名牌、气泡均不被裁掉。
- 四只猫同时在工作亭、两只猫同时在树下时仍能辨认账号。
- 白天、黄昏、夜晚切换不移动任何交互设施。
- 晴、云、雨、雪都不会降低名牌和错误提示的可读性。
- 拖起、移动、放下时角色锚点不跳，命中高亮与视觉设施一致。
- 工作亭、邮筒、任务道、池塘、瞭望点、草坪六处行为与本文表格一致。
- 关闭动画偏好时仍有清晰静态状态，不依赖闪烁传达唯一信息。
- 缺一张图或加载失败时可回退，不出现白屏或无法选择账号。
- 画师提出的逻辑修订已同步到 `interactions.js` / `cats.js` / 文档，不能只改图不改命中。
